# 基于 Pytest 的电商接口自动化测试框架接口文档

## 项目说明

本项目用于测试实习准备，目标是搭建一个小型电商接口 Demo，并围绕登录、商品、购物车、订单等核心业务模块完成接口测试和接口自动化测试。

项目技术栈：

```text
后端 Demo：Python + Flask + SQLite
接口管理：Apifox
自动化测试：Requests + Pytest
测试报告：Allure
版本管理：Git
```

第一阶段先设计登录接口。登录接口是后续鉴权、购物车、订单接口的基础。

## 通用响应结构

所有接口统一返回 JSON。

成功响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

失败响应结构：

```json
{
  "code": 1001,
  "message": "username or password error",
  "data": null
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| code | integer | 是 | 业务状态码，0 表示成功，非 0 表示失败 |
| message | string | 是 | 响应提示信息 |
| data | object/null | 是 | 业务数据，失败时可以为 null |

## 错误码约定

| 错误码 | 含义 | 使用场景 |
|---:|---|---|
| 0 | success | 请求处理成功 |
| 1001 | username or password error | 用户名或密码错误 |
| 1002 | username is required | 用户名为空或缺失 |
| 1003 | password is required | 密码为空或缺失 |
| 1004 | user is locked | 用户已被锁定 |
| 1005 | invalid content type | 请求头 Content-Type 不正确 |
| 1006 | invalid json body | 请求体不是合法 JSON |

## 接口 1：用户登录

### 基本信息

| 项目 | 内容 |
|---|---|
| 接口名称 | 用户登录 |
| 请求方法 | POST |
| 接口地址 | /api/login |
| 接口说明 | 用户输入用户名和密码，校验通过后返回 token |
| 是否需要登录 | 否 |
| Content-Type | application/json |

### 请求 Headers

| Header | 是否必填 | 示例值 | 说明 |
|---|---:|---|---|
| Content-Type | 是 | application/json | 表示请求体为 JSON |

### 请求 Body

```json
{
  "username": "standard_user",
  "password": "secret_sauce"
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 规则 |
|---|---|---:|---|
| username | string | 是 | 用户名不能为空 |
| password | string | 是 | 密码不能为空 |

### 测试账号

| 用户名 | 密码 | 状态 | 说明 |
|---|---|---|---|
| standard_user | secret_sauce | 正常 | 可正常登录 |
| locked_out_user | secret_sauce | 锁定 | 不允许登录 |

### 成功响应

HTTP 状态码：

```text
200 OK
```

响应体：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "mock-token-standard_user",
    "username": "standard_user"
  }
}
```

成功断言：

```text
1. HTTP 状态码等于 200。
2. 响应体 code 等于 0。
3. 响应体 message 等于 success。
4. data 不为空。
5. data.token 存在且不为空。
6. data.username 等于请求中的 username。
```

### 失败响应：密码错误

HTTP 状态码：

```text
200 OK
```

响应体：

```json
{
  "code": 1001,
  "message": "username or password error",
  "data": null
}
```

断言重点：

```text
1. code 不等于 0。
2. message 明确说明用户名或密码错误。
3. data 为 null。
4. 不能返回 token。
```

### 失败响应：用户名为空

```json
{
  "code": 1002,
  "message": "username is required",
  "data": null
}
```

### 失败响应：密码为空

```json
{
  "code": 1003,
  "message": "password is required",
  "data": null
}
```

### 失败响应：用户被锁定

```json
{
  "code": 1004,
  "message": "user is locked",
  "data": null
}
```

### 失败响应：Content-Type 错误

```json
{
  "code": 1005,
  "message": "invalid content type",
  "data": null
}
```

## 登录接口测试点

### 正常场景

```text
1. 正确用户名和正确密码登录成功。
2. 登录成功后返回 token。
3. 登录成功后返回的 username 与请求 username 一致。
```

### 异常场景

```text
1. 正确用户名和错误密码登录失败。
2. 用户名为空时登录失败。
3. 密码为空时登录失败。
4. username 字段缺失时登录失败。
5. password 字段缺失时登录失败。
6. 用户名不存在时登录失败。
7. 被锁定用户登录失败。
8. 请求体不是合法 JSON 时登录失败。
9. Content-Type 不是 application/json 时登录失败。
```

### 安全场景

```text
1. 登录失败时不能返回 token。
2. 密码不能出现在响应体中。
3. 用户名或密码错误时，不应暴露过多账号状态信息。
4. 被锁定账号不能登录成功。
```

### 自动化测试优先级

第一批自动化用例先覆盖：

```text
1. 正确账号密码登录成功。
2. 错误密码登录失败。
3. 用户名为空。
4. 密码为空。
5. 被锁定用户登录失败。
6. 登录失败不返回 token。
```
